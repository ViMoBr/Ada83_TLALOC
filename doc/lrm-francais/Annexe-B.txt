
                     B. Pragmas de langage prédéfinis  

Cette annexe définit les pragmas LIST, PAGE et OPTIMIZE, et résume les
définitions données ailleurs des pragmas restants définis par le langage.

Pragma Signification

CONTROLLED Prend le nom simple d'un type d'accès comme unique
              argument. Ce pragma n'est autorisé qu'immédiatement dans
              la partie déclarative ou la spécification de package qui contient
              la déclaration du type d'accès ; la déclaration doit
              se produire avant le pragma. Ce pragma n'est pas autorisé pour un
              type dérivé. Ce pragma précise que le stockage automatique
              la remise en état ne doit pas être effectuée pour les objets désignés par
              valeurs du type d'accès, sauf en quittant le
              instruction de bloc, corps de sous-programme ou corps de tâche qui contient
              la déclaration du type d'accès, ou après avoir quitté le
              programme (voir 4.8).

ELABORATE Prend un ou plusieurs noms simples désignant les unités de bibliothèque comme
              arguments. Ce pragma n'est autorisé qu'immédiatement après
              la clause de contexte d'une unité de compilation (avant le
              unité de bibliothèque ultérieure ou unité secondaire). Chaque argument
              doit être le nom simple d'une unité de bibliothèque mentionnée par le
              clause de contexte. Ce pragma précise que le
              le corps de l'unité de bibliothèque correspondant doit être élaboré avant
              l'unité de compilation donnée. Si l'unité de compilation donnée
              est une sous-unité, le corps de l'unité de bibliothèque doit être élaboré
              avant le corps de l'unité de bibliothèque ancêtre de la sous-unité
              (voir 10.5).

INLINE Prend un ou plusieurs noms comme arguments ; chaque nom est soit
              le nom d'un sous-programme ou le nom d'un générique
              sous-programme. Ce pragma n'est autorisé qu'à l'endroit d'un
              élément déclaratif dans une partie ou un package déclaratif
              spécification, ou après une unité de bibliothèque dans une compilation, mais
              avant toute unité de compilation ultérieure. Ce pragme
              précise que les organes du sous-programme doivent être élargis
              en ligne à chaque appel dans la mesure du possible ; dans le cas d'un
              sous-programme générique, le pragma s'applique aux appels de son
              instanciations (voir 6.3.2).

INTERFACE Prend un nom de langage et un nom de sous-programme comme arguments.
              Ce pragma est autorisé à la place d'un élément déclaratif,
              et doit s'appliquer dans ce cas à un sous-programme déclaré par un
              élément déclaratif antérieur de la même partie déclarative ou
              spécification du paquet. Ce pragma est également autorisé pour un
              unité de bibliothèque; dans ce cas le pragma doit apparaître après le 
              déclaration de sous-programme, et avant toute déclaration ultérieure
              unité de compilation. Ce pragma précise l'autre langue
              (et donc les conventions d'appel) et informe le
              compilateur qu'un module objet sera fourni pour le
              sous-programme correspondant (voir 13.9).

LISTE Prend l'un des identifiants ON ou OFF comme unique
              argument. Ce pragma est autorisé partout où se trouve un pragma
              autorisé. Il précise que le listing de la compilation est à
              être poursuivi ou suspendu jusqu'à ce qu'un pragma LIST avec le 
              l’argument opposé est donné dans la même compilation. Le
              pragma lui-même est toujours répertorié si le compilateur produit
              une liste.

MEMORY_SIZE Prend un littéral numérique comme argument unique. Ce pragme
              n'est autorisé qu'au début d'une compilation, avant le
              première unité de compilation (le cas échéant) de la compilation. Le
              L'effet de ce pragma est d'utiliser la valeur du spécifié
              littéral numérique pour la définition du nombre nommé
              MEMORY_SIZE (voir 13.7).                                    

OPTIMIZE Prend comme unique l'un des identifiants TIME ou SPACE.
              argument. Ce pragma n'est autorisé que dans un cadre déclaratif
              partie et elle s'applique au bloc ou au corps renfermant le
              partie déclarative. Il précise si le temps ou l'espace est le
              critère principal d’optimisation.

PACK Prend le nom simple d'un type d'enregistrement ou de tableau comme
              un seul argument. Les positions autorisées pour ce pragma, et
              les restrictions sur le type nommé sont régies par le même
              règles comme pour une clause de représentation. Le pragma précise
              que la minimisation du stockage devrait être le critère principal lorsque
              sélectionner la représentation du type donné (voir 13.1).

PAGE Ce pragma n'a aucun argument et est autorisé partout où un
              le pragma est autorisé. Il précise que le texte du programme qui
              suit, le pragma devrait commencer sur une nouvelle page (si le
              le compilateur produit actuellement une liste). 

PRIORITY Prend une expression statique du sous-type entier prédéfini
              PRIORITY comme argument unique. Ce pragma est seulement
              autorisé dans le cadre de la spécification d'une unité de tâche ou
              immédiatement dans la partie déclarative la plus externe d'un message principal
              programme. Il précise la priorité de la tâche (ou des tâches de
              le type de tâche) ou la priorité du programme principal (voir
              9.8).

SHARED Prend le nom simple d'une variable comme argument unique.
              Ce pragma n'est autorisé que pour une variable déclarée par un
              déclaration d'objet et dont le type est un scalaire ou un accès
              taper; la déclaration de variable et le pragma doivent tous deux
              se produire (dans cet ordre) immédiatement dans le même
              partie déclarative ou spécification de package. Ce pragme
              spécifie que chaque lecture ou mise à jour de la variable est un
              point de synchronisation pour cette variable. Une mise en œuvre
              doit restreindre les objets pour lesquels ce pragma est autorisé
              à des objets pour lesquels chacun de lecture directe et direct
              la mise à jour est réalisée comme une opération indivisible (voir
              9.11).

STORAGE_UNIT Prend un littéral numérique comme argument unique. Ce pragme
              n'est autorisé qu'au début d'une compilation, avant le
              première unité de compilation (le cas échéant) de la compilation. Le
              L'effet de ce pragma est d'utiliser la valeur du spécifié
              littéral numérique pour la définition du nombre nommé
              STORAGE_UNIT (voir 13.7).

SUPPRESS Prend en arguments l'identifiant d'un chèque et éventuellement
              également le nom soit d'un objet, soit d'un type ou d'un sous-type, d'un
              sous-programme, une unité de tâche ou une unité générique. Ce pragme est
              uniquement autorisé soit immédiatement dans une partie déclarative, soit
              immédiatement dans une spécification de package. Dans ce dernier
              Dans ce cas, la seule forme autorisée est celle avec un nom qui désigne un
              entité (ou plusieurs sous-programmes surchargés) déclarée
              immédiatement dans les spécifications du package. Le
              la permission d'omettre le chèque donné s'étend du lieu de 
              le pragma à la fin de la région déclarative associée
              avec l'instruction de bloc ou le programme englobant le plus à l'intérieur
              unité. Pour un pragma donné dans une spécification de package, le
              l'autorisation s'étend jusqu'à la fin de la portée du nom
              entité.

              Si le pragma inclut un nom, l'autorisation d'omettre le
              un chèque donné est encore plus restreint : il n'est donné que pour
              opérations sur l'objet nommé ou sur tous les objets de la base
              type d'un type ou d'un sous-type nommé ; pour les appels d'un nommé
              sous-programme ; pour les activations des tâches de la tâche nommée
              taper; ou pour les instanciations de l'unité générique donnée (voir
              11.7).

SYSTEM_NAME Prend un littéral d'énumération comme argument unique. Ce
              pragma n'est autorisé qu'au début d'une compilation, avant
              la première unité de compilation (le cas échéant) de la compilation. Le
              L'effet de ce pragma est d'utiliser le littéral d'énumération avec
              l'identifiant spécifié pour la définition de la constante
              SYSTEM_NAME. Ce pragma n'est autorisé que si le spécifié
              l'identifiant correspond à l'un des littéraux du type
              NOM déclaré dans le package SYSTEM (voir 13.7).
